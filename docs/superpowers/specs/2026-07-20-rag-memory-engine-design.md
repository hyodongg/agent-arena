# RAG 메모리 엔진 설계 (2026-07-20)

제품 방향 로드맵의 **서브프로젝트 5**, 마지막 조각. 에이전트가 과거 매매의 성공/실패를 기억하고 유사한 뉴스를 만났을 때 회상하게 한다.

## 목표

지금 에이전트는 매번 백지에서 판단한다. 같은 종목에 같은 유형의 뉴스가 다시 떠도 지난번에 그 판단으로 벌었는지 잃었는지 모른다.

과거 매매를 "그때 이런 뉴스에 이렇게 했더니 이만큼 벌었다/잃었다"는 기억으로 남기고, 새 뉴스가 오면 **의미가 비슷한 과거 기억을 검색해 프롬프트에 넣어준다.**

## 설계

### 1. 기억의 단위

매매 하나가 기억 하나다. `AgentMemory`:

- `agent`, `stock`
- `newsTitle` — 그때의 뉴스 제목
- `action` — BUY / SELL
- `executionPrice` — 체결가
- `embedding` — 뉴스 제목의 임베딩 벡터
- `returnRate` — 성과. 평가 전에는 null이다
- `evaluatedAt`

### 2. 성과는 나중에 채운다

매매 직후에는 벌었는지 잃었는지 알 수 없다. **`MemoryEvaluationScheduler`가 체결 후 10분이 지난 미평가 기억을 찾아 현재가와 비교해 수익률을 기록한다.**

- BUY: `(현재가 - 체결가) / 체결가`
- SELL: `(체결가 - 현재가) / 체결가`

평가가 끝난 기억만 회상 대상이 된다. 성과를 모르는 기억은 주입해봐야 도움이 안 된다.

### 3. 회상

`AgentDecisionMaker`가 결정을 내리기 전에, 에이전트별로 **현재 뉴스 제목과 의미가 가장 가까운 과거 기억 3건**을 찾아 프롬프트에 넣는다.

유사도는 코사인 유사도다. 임베딩은 `text-embedding-3-small`에 `dimensions=256`을 줘서 받는다. 1536차원을 그대로 쓰면 행당 저장 용량이 6배가 되는데, 이 용도에는 256으로 충분하다.

### 4. 벡터 저장은 MySQL에 한다

원래 로드맵(`backend/CLAUDE.md` 4단계)은 벡터 저장소를 별도로 분리한다고 적어뒀다. **지금은 그렇게 하지 않는다.**

임베딩을 `AgentMemory` 행의 TEXT 컬럼에 JSON 배열로 저장하고, 코사인 유사도는 자바에서 전체 스캔으로 계산한다. 기억이 수백 건 수준이면 이게 가장 단순하고, 별도 인프라를 띄울 이유가 없다.

이 프로젝트의 원칙은 "문제를 먼저 겪고, 그 문제를 근거로 다음 기술을 도입한다"이다. 전체 스캔이 실제로 느려지는 걸 확인한 뒤에 전용 벡터 저장소로 옮기는 것이 순서에 맞다. **이 지점은 직접 부하를 걸어 실험해볼 대상이다** (아래 "부하 실험 지점" 참고).

### 5. 새로 만드는 것

- `domain/agent/AgentMemory` — 엔티티
- `domain/agent/repository/AgentMemoryRepository`
- `common/client/openai/OpenAiClient.embed()` — 임베딩 발급
- `domain/agent/service/AgentMemoryService` — 기억 생성, 유사 기억 검색
- `domain/agent/scheduler/MemoryEvaluationScheduler` — 미평가 기억의 수익률 계산
- `AgentDecisionMaker` — 프롬프트에 회상 결과 주입
- `OrderServiceImpl` — 체결 시 기억 생성

### 6. 이번 범위에 넣지 않는 것

- 전용 벡터 저장소(Qdrant/pgvector) — 위 4번 참고
- 임베딩 캐싱 — 같은 뉴스 제목이 반복되지 않는다
- 기억 요약/압축 — 기억이 쌓여 프롬프트가 길어지는 문제를 아직 겪지 않았다
- 폴백 시(LLM 장애) 회상 — 폴백은 원래 확률 규칙이라 기억을 쓰지 않는다

## 검증 방법

1. 매매가 일어나면 `agent_memories` 행이 생기고 임베딩이 채워지는지
2. 10분 뒤 스케줄러가 `returnRate`를 채우는지, BUY/SELL 부호가 맞는지
3. 회상이 실제로 유사한 기억을 찾는지 — 관련 없는 뉴스가 상위로 오면 안 된다
4. 프롬프트에 회상 결과가 들어간 상태에서 결정이 나오는지
5. 기억이 없는 초기 상태에서도 정상 동작하는지

## 부하 실험 지점

구현 후 직접 테스트해볼 만한 지점이다. 미리 해결하지 않는다.

- 유사도 검색이 매 뉴스마다 `agent_memories` 전체를 읽고 자바에서 계산한다. 기억이 쌓일수록 선형으로 느려진다.
- `agent_memories`를 `agent_id` + `evaluated_at IS NOT NULL`로 조회하는데 인덱스가 없다.
- 임베딩 TEXT 컬럼이 행마다 수 KB라 풀스캔 시 I/O가 커진다.
