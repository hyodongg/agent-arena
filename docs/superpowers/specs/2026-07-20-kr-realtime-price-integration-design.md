# 국내 주식 실시세 연동 설계 (2026-07-20)

제품 방향 로드맵(`2026-07-19-product-direction-roadmap.md`)의 **서브프로젝트 1** 중 국내(KRX) 부분. 미국 주식은 이번 범위에 없다.

## 목표

`Stock.currentPrice`를 에이전트 매매가 밀어올리는 값에서, 한국투자증권(KIS) API로 동기화되는 실제 시장 가격으로 바꾼다. 에이전트는 이 가격을 참조해 가상으로 체결할 뿐, 더 이상 가격을 움직이지 않는다.

## 현재 구조의 문제

`OrderServiceImpl.simulateForNews()`는 체결마다 `BUY_IMPACT`(1.01) / `SELL_IMPACT`(0.99)를 곱해 `stock.reflectTrade()`로 가격을 덮어쓴다. 에이전트가 사면 가격이 오르고, 오른 가격으로 보유분이 재평가되어 수익률이 부풀고, 또 사면 더 오르는 자기충족 루프다. 실제로 로컬 데이터에서 누적수익률 1952%가 관측됐다.

실시세를 도입하면 이 루프가 끊긴다. 수익률이 현실적인 수준으로 줄어들면서 에이전트 간 차이도 작아지는데, **차별화 장치(레버리지 등)는 이번 범위에 넣지 않는다.** `AgentStyle`의 반응확률·베팅배율 차이만으로 얼마나 갈리는지 실제로 돌려본 뒤 판단한다.

## 실측한 API 제약

모의투자(VTS) 환경에서 직접 측정한 값이다.

| 항목 | 값 | 근거 |
|---|---|---|
| 시세 조회 | **초당 2~3건** | 30종목 무지연 연속 호출 시 3초간 성공 8 / 실패 22 (`초당 거래건수를 초과하였습니다`) |
| 토큰 발급 | **1분당 1회** | 재발급 시 `EGW00133: 접근토큰 발급 잠시 후 다시 시도하세요(1분당 1회)` |
| 토큰 유효기간 | 24시간 | 발급 응답의 `access_token_token_expired` |

모의투자 앱키로 시세 조회가 정상 동작함을 확인했다(삼성전자 실시간 가격 수신). **실전투자 앱키는 필요 없다.**

조회 응답(`output`)에서 사용하는 필드:
- `stck_prpr` — 현재가
- `acml_vol` — 누적거래량

종목명(`hts_kor_isnm`)은 이 응답에 없으므로 시드할 때 직접 넣는다.

## 설계

### 1. Stock을 "시장 데이터 미러"로 정리

`Stock`은 이제 외부 시장 데이터를 비추는 역할만 한다. 값을 바꾸는 주체는 동기화 스케줄러 하나뿐이다.

```
- public void reflectTrade(BigDecimal executedPrice, BigDecimal tradedQuantity)
+ public void syncMarketData(BigDecimal currentPrice, BigDecimal volume)
```

`volume`도 아레나 내부 누적 체결량이 아니라 실제 시장 누적거래량이 된다.

경계가 이렇게 갈린다:
- **Stock** — 시장 데이터. 스케줄러만 쓴다.
- **Order / AgentHolding / Agent** — 아레나 시뮬레이션. 시세를 읽기만 한다.

### 2. `KisClient` 신설

위치: `common/client/kis/` (도메인이 아니라 외부 연동이므로 `domain/` 밖)

- `KisProperties` — `@ConfigurationProperties(prefix = "kis")`로 base-url / app-key / app-secret 주입
- `KisClient.fetchQuote(String stockCode)` → `KisQuote(BigDecimal currentPrice, BigDecimal volume)`
- HTTP는 `RestClient` 사용
- **토큰 캐싱**: 메모리에 토큰과 만료시각을 들고 있다가, 만료 10분 전부터 재발급. 1분 쿨다운에 걸리면 예외를 던지지 않고 해당 사이클을 건너뛴다.

토큰 캐시를 Redis가 아닌 메모리에 두는 건 의도적이다. 인스턴스가 하나뿐이라 분산 캐시가 풀 문제가 아직 없다.

### 3. `StockPriceSyncScheduler` 신설

위치: `domain/stock/scheduler/`

- `@Scheduled(fixedDelay = 5000)` — 한 바퀴가 끝난 뒤 5초 쉬고 다시 시작
- **장중에만 동작**: `Asia/Seoul` 기준 평일 09:00~15:30. 그 외에는 아무것도 하지 않고 즉시 반환
- 종목을 **순차로** 조회하며 **종목당 500ms 간격**(초당 2건, 실측 한도 내). 30종목 한 바퀴 약 15초
- 개별 종목 조회가 실패하면 **그 종목만 건너뛰고** 마지막 값을 유지한다. 로그를 남기되 스케줄러는 죽지 않는다
- 공휴일은 판별하지 않는다. 휴장일에 호출해도 값이 안 바뀔 뿐이라 무해하다

15초 동안 스케줄러 스레드가 `Thread.sleep`으로 묶이는 건 알고 있는 약점이다. 지금은 그대로 두고, 나중에 부하를 걸어 직접 병목을 확인한 뒤 근거를 갖고 개선한다.

### 4. `OrderServiceImpl` 수정

- `BUY_IMPACT` / `SELL_IMPACT` 상수 삭제
- `stock.reflectTrade(...)` 호출 삭제
- 체결가는 `stock.getCurrentPrice()`를 그대로 사용 (기존과 동일)

나머지(반응 확률 판정, 명목금액 기반 수량 계산, 현금·보유수량 정산, 누적수익률 재계산)는 건드리지 않는다.

### 6. 임시 뉴스 시드

뉴스가 없으면 에이전트가 매매를 안 해서 연동이 동작하는지 확인할 수 없다. 코스피 종목을 참조하는 더미 뉴스 **20건**(호재 10 / 악재 10, 서로 다른 종목에 분산)을 `backend/scripts/seed-dummy-news.sh`로 시드한다. **이 더미 뉴스는 서브프로젝트 2(실뉴스 연동)에서 통째로 교체된다.**

### 7. 이번 범위에 넣지 않는 것

- `market` / `currency` 필드 — 미국 주식을 붙일 때 추가한다
- WebSocket 실시간 구독 — REST 폴링으로 충분하다
- 재시도 / 지수 백오프 — rate limit을 실제로 겪어본 뒤에 판단한다
- Redis 토큰 캐시 — 인스턴스가 하나다
- 에이전트 수익률 차별화 장치 — 위 "현재 구조의 문제" 참고

## 설정

`application-local.yaml`에 추가:

```yaml
kis:
  base-url: https://openapivts.koreainvestment.com:29443
  app-key: ${KIS_APP_KEY}
  app-secret: ${KIS_APP_SECRET}
```

`KIS_APP_KEY` / `KIS_APP_SECRET`은 `backend/.env`에 있다(gitignore됨, `spring.config.import`로 자동 로딩).

## 검증 방법

장중(평일 09:00~15:30)에:

1. 시드 스크립트를 돌려 30종목이 실제 가격으로 등록되는지 확인
2. 앱을 띄우고 로그에서 동기화 사이클이 도는지, rate limit 에러가 안 나는지 확인
3. `GET /api/stocks`를 시간차를 두고 두 번 호출해 가격이 실제로 변하는지 확인 — 네이버 금융 등 외부 시세와 대조
4. 뉴스 주입 → 주문 생성 후, 해당 종목 가격이 **체결 때문에 변하지 않는지** 확인 (기존엔 1%씩 튀었음)
5. 프론트엔드 `/arena`에서 종목 테이블에 실제 코스피 가격이 뜨는지 확인

휴장 중에는 3번 대신 "동기화가 스킵되고 가격이 고정되는지"를 확인한다.

## 알려진 부하/동시성 실험 지점

구현 후 직접 테스트해볼 만한 지점들이다. 이번 범위에서 미리 해결하지 않는다.

- 스케줄러 스레드가 한 사이클 15초를 `Thread.sleep`으로 점유한다
- 종목 수를 늘리면 한 바퀴 도는 시간이 선형으로 늘어난다
- 동기화 트랜잭션과 뉴스 주입 트랜잭션이 같은 `stocks` 행을 동시에 건드릴 수 있다
- `agent_holdings`는 `(agent_id, stock_id)`로 조회하는데 인덱스가 없다
