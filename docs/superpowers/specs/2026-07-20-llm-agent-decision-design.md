# LLM 기반 에이전트 의사결정 설계 (2026-07-20)

제품 방향 로드맵의 **서브프로젝트 4**. 키워드 규칙을 LLM 판단으로 바꾼다.

## 현재 구조의 한계

`AgentStyle.classify()`는 `investmentPrompt`에 "단타"/"장기"/"공격"/"보수"가 들어있는지만 본다. 5종 중 하나로 분류되고 나면 프롬프트의 나머지 내용은 버려진다. 같은 스타일로 분류된 에이전트는 완전히 동일하게 행동한다.

`OrderServiceImpl.simulateForNews()`도 뉴스의 호재/악재만 보고 방향을 정한다. 호재면 무조건 매수, 악재면 무조건 매도다. 보유하지 않은 종목도 매도하고, 현금이 없어도 매수한다.

## 설계

### 1. 뉴스 하나당 LLM 호출 한 번

에이전트마다 따로 물으면 뉴스 한 건에 N번 호출된다. **모든 에이전트의 성향과 포트폴리오를 한 프롬프트에 넣고 한 번에 결정을 받는다.**

뉴스 주입은 5초 주기지만 수집이 그보다 느리므로, 실제 호출량은 뉴스 생산 속도(분당 3건 내외)를 넘지 않는다.

### 2. LLM에 주는 정보

- 뉴스 제목, 관련 종목명, 호재/악재 판정, 현재가
- 에이전트별: 이름, `investmentPrompt` 원문, 현금잔고, 해당 종목 보유수량

`investmentPrompt`를 요약하지 않고 그대로 넘긴다. 사용자가 쓴 지침이 그대로 반영되는 것이 이 서브프로젝트의 목적이다.

### 3. LLM이 돌려주는 것

에이전트별로 `action`(BUY/SELL/HOLD)과 `conviction`(1~5)이다.

- 매매 금액 = 100,000원 × conviction (10만~50만원)
- HOLD면 주문을 만들지 않는다
- 보유수량이 0인데 SELL이면 무시한다 — 공매도는 이 시뮬레이션 범위가 아니다

현금 부족은 검사하지 않는다. 현금이 마이너스로 갈 수 있는 건 기존 동작 그대로다.

### 4. 실패 시 폴백

LLM 호출이 실패하면 기존 `AgentStyle` 확률 규칙으로 매매한다. 외부 API 장애 때문에 아레나가 통째로 멈추면 안 된다. `AgentStyle`은 이 폴백 경로에서 계속 쓰이므로 남겨둔다.

### 5. 새로 만드는 것

- `domain/agent/service/AgentDecision` — `(agentId, action, conviction)` 레코드
- `domain/agent/service/AgentDecisionMaker` — 프롬프트 구성, LLM 호출, 응답 파싱
- `OrderServiceImpl.simulateForNews()` 수정 — 주사위 굴리기를 LLM 결정으로 교체, 실패 시 폴백

### 6. 이번 범위에 넣지 않는 것

- 과거 매매 기억 주입 — 서브프로젝트 5(RAG)의 범위다
- 종목 선택 — 에이전트는 여전히 뉴스가 지목한 종목만 거래한다
- 공매도, 현금 제약, 수수료

## 검증 방법

1. 성향이 다른 에이전트가 같은 뉴스에 다르게 반응하는지 (특히 "안전제일봇" vs "몰빵도전자")
2. HOLD 결정이 실제로 주문을 만들지 않는지
3. 보유수량 0인 에이전트가 SELL 판정을 받아도 주문이 안 만들어지는지
4. LLM 호출 실패 시 폴백이 동작하는지
5. 뉴스 하나당 LLM 호출이 1회인지 (에이전트 수만큼이 아니라)
